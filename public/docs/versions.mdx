# Versions & Release Policy

## Current version

The latest stable release is always published to npm under the `latest` tag:

```bash
npm install -g @prokodo/cli@latest
```

Check which version you have installed:

```bash
prokodo --version
```

## Version scheme

`@prokodo/cli` follows [Semantic Versioning](https://semver.org):

| Increment         | When                                                    |
| ----------------- | ------------------------------------------------------- |
| **patch** `0.x.Y` | Bug fixes, documentation updates, CI/infra changes      |
| **minor** `0.X.0` | New commands or flags that are backwards-compatible     |
| **major** `X.0.0` | Breaking changes to command signatures or config format |

## Release process

Releases are fully automated via GitHub Actions:

1. A maintainer triggers the **Bump version** workflow from the Actions tab and selects `patch`, `minor`, or `major`.
2. The workflow increments `package.json`, prepends a new section to the [Changelog](./changelog), commits the change, and pushes a `vX.Y.Z` git tag.
3. The **Release** workflow detects the new tag, runs the full test suite, and publishes to npm with `pnpm publish`.
4. The docs are rebuilt and deployed automatically.

## Installing a specific version

```bash
# Install a specific version globally
npm install -g @prokodo/cli@0.1.0

# Or with pnpm
pnpm add -g @prokodo/cli@0.1.0
```

## Viewing all published versions

```bash
npm view @prokodo/cli versions --json
```

Or browse directly on [npmjs.com](https://www.npmjs.com/package/@prokodo/cli?activeTab=versions).

## Upgrading

```bash
# npm
npm update -g @prokodo/cli

# pnpm
pnpm update -g @prokodo/cli
```

After upgrading, run `prokodo doctor` to verify the new version is active and the environment is healthy.

<Admonition type="info" title="Breaking changes">
  Any breaking changes are clearly marked in the [Changelog](./changelog) and the major version is
  bumped. Check the changelog before upgrading across major versions.
</Admonition>
